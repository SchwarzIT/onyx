Our design system utilizes column grids to establish structured layouts that organize content, components, and UI elements into a cohesive hierarchy. This framework ensures consistent alignment and distribution across all applications. By acting as the foundation for responsive behavior, the grid enables seamless adaptation across various screen sizes and resolutions, ultimately enhancing usability and the overall user experience.

::info-card{headline="Explore the Grid" icon="grid"}
Visit our interactive [Grid Playground](https://storybook.onyx.schwarz/?path=/story/examples-gridplayground--default) to see our grid system in action.
::

## Grid

Our grid system is basically build out of the following three main building blocks:

<steps>

::step
#headline
Columns

#default
The available width (either the whole screen or only a specific section / element) is divided into up to 12 columns.
Each element inside the grid can then define how many columns it should span. The grid will then take care of positioning the elements accordingly and wrap them into new rows if they exceed the available columns.

The maximum number of available columns per row will adapt dynamically based on the screen / container breakpoint.
This adaptive approach ensures out-of-the-box responsive behavior of the elements and a consistent width of elements across different screen sizes, pages and applications. See the [breakpoints](#breakpoints) below for an overview of the available breakpoints and their maximum grid columns.
::

::step
#headline
Gutters / Gaps

#default
Elements placed inside the grid are automatically spaced with pre-defined gutters / gaps.
::

::step
#headline
Margins

#default
Optionally, unified margins can be placed around the grid which is particularly useful when applied to the whole page content so every page has consistent and dynamic spacing around the page content so it does not "stick" to the screen borders.
::

</steps>

### Usage

::content-tabs
#dev
To create a new grid container, simply add the `onyx-grid` CSS class to any element. All elements within are then able to define how many columns they should span.

```vue [Example grid usage]
<template>
  <div class="onyx-grid">
    <OnyxCard class="onyx-grid-span-4">Grid element 1</OnyxCard>
    <OnyxCard class="onyx-grid-span-4">Grid element 2</OnyxCard>
    <OnyxCard class="onyx-grid-span-4">Grid element 3</OnyxCard>
  </div>
</template>
```

We support the following CSS classes to define how many columns an element should span:

<div class="onyx-grid">

  ::details-card
  ---
  name: default
  items:
    - label: Description
      value: Spans <number> of columns
    - label: CSS class
      value: onyx-grid-span-<number>
  ---
  ::

  ::details-card
  ---
  name: full width
  items:
    - label: Description
      value: Spans the whole row
    - label: CSS class
      value: onyx-grid-span-full
      copy: true
  ---
  ::

  ::details-card
  ---
  name: specific breakpoints
  items:
    - label: Description
      value: Spans <number> of columns for the specific breakpoint and higher
    - label: CSS class
      value: onyx-grid-<breakpoint>-span-<number>
  ---
  ::

</div>

<br />

The outer grid margin is already added by our [OnyxPageLayout](https://storybook.onyx.schwarz/?path=/docs/layout-pagelayout--docs) component. If you still need to implement it manually, simply use the `.onyx-grid-layout` CSS class:

```vue [Manually apply grid margin]
<template>
  <OnyxPageLayout no-padding>
    <!-- place e.g. full width content like a hero here...  -->

    <div class="onyx-grid-layout">
      <!-- place your page content here -->
    </div>
  </OnyxPageLayout>
</template>
```

### Customization

You can customize the grid in several ways as shown below. Simply set the corresponding CSS classes on the root element of your application.

<steps>

::step
#headline
Max width

#default
By default, components can take up the whole available screen width. Since content can be overwhelming for users with larger screens such as 2k and beyond, you can limit the overall grid content width of the application. Grid usages and components such as the OnyxNavBar or OnyxBottomBar will adapt automatically.

To limit the width, apply one of the following options to your root application element:

<div class="onyx-grid">

  ::details-card
  ---
  name: none
  items:
    - label: Max. width
      value: none
    - label: CSS class
      value: none (default)
  ---
  ::

  ::details-card
  ---
  name: md
  items:
    - label: Max. width
      value: 1440px
    - label: CSS class
      value: onyx-grid-max-md
      copy: true
  ---
  ::

  ::details-card
  ---
  name: lg
  items:
    - label: Max. width
      value: 1920px
    - label: CSS class
      value: onyx-grid-max-lg
      copy: true
  ---
  ::

</div>

::

::step
#headline
Alignment

#default
The grid alignment is only relevant when you have set a max width (see above). In this case you can decide whether the grid should be left or center aligned when the users screens exceeds the max width:

<div class="onyx-grid">

  ::details-card
  ---
  name: left
  items:
    - label: Alignment
      value: left
    - label: CSS class
      value: none (default)
  ---
  ::

  ::details-card
  ---
  name: center
  items:
    - label: Alignment
      value: center
    - label: CSS class
      value: onyx-grid-center
      copy: true
  ---
  ::

</div>

::

::step
#headline
Max columns

#default
The grid is limited to 12 columns by default. To make use of extra space available on larger screens such as 4k monitors, you can optionally increase the maximum number of columns to either 16 or 20. Note that the extra columns are only available for the `lg` breakpoint and above.

  ::

  ---

  items:
    - label: Max. columns
      value: 16
    - label: CSS class
      value: onyx-grid-lg-16
      copy: true
  ---
  ::

  ::details-card
  ---
  name: 20
  items:
    - label: Max. columns
      value: 20
    - label: CSS class
      value: onyx-grid-xl-20
      copy: true
  ---
  ::

</div>

::

</steps>

## Breakpoints

Breakpoints are specific widths where the page or component layout automatically changes into a different layout so its stays readable on a smaller or larger device. Our design system defines the following breakpoints:

<div class="onyx-grid">

::details-card
---
name: 2xs
items:
  - label: Width
    value: 320px to 576px
  - label: Max. grid columns
    value: 4
---
::

::details-card
---
name: xs
items:
  - label: Width
    value: 577px to 768px
  - label: Max. grid columns
    value: 8
---
::

::details-card
---
name: sm
items:
  - label: Width
    value: 769px to 992px
  - label: Max. grid columns
    value: 8
---
::

::details-card
---
name: md
items:
  - label: Width
    value: 993px to 1440px
  - label: Max. grid columns
    value: 12
---
::

::details-card
---
name: lg
items:
  - label: Width
    value: 1441px to 1920px
  - label: Max. grid columns
    value: 12 or 16
---
::

::details-card
---
name: xl
items:
  - label: Width
    value: 1921px and beyond
  - label: Max. grid columns
    value: 12, 16 or 20
---
::

</div>

Our smallest supported breakpoints is 320px which means that we do not guarantee optimized layouts for smaller screens.

### Usage

::content-tabs
#dev
When using custom layouts or components, you can use our SCSS mixins to easily apply styles when either the whole screen or an container / element width matches a specific min/max breakpoint:

<steps>

  ::step
  #headline
  Media queries

  #default
  To apply styles only when the **whole screen** has a specific min or max width, use our `breakpoints.screen()` SCSS mixin that applies [CSS media queries](https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Media_queries/Using).

  ```scss [Applying styles for specific screen breakpoints]
  @use "sit-onyx/breakpoints.scss";

  .some-class {
    @include breakpoints.screen(max, sm) {
      // your styles for md screens and smaller
    }

    @include breakpoints.screen(min, md) {
      // your styles for md screens and larger
    }
  }
  ```

  ::

  ::step
  #headline
  Container queries

  #default
  To apply styles only when a **single element** has a specific min or max width, use our `breakpoints.container()` SCSS mixin that applies [CSS container queries](https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Containment/Container_queries). Before applying the container queries, ensure the add the [container-type](https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Containment/Container_queries#using_container_queries) CSS property.

  The breakpoint is considered inclusively so when defining min and max for the same breakpoint, make sure to add an offset for one of them.

  ```scss [Applying styles for specific container breakpoints]
  @use "sit-onyx/breakpoints.scss";

  .some-class {
    container-type: size;

    @include breakpoints.container(max, md) {
      // your styles for md containers and smaller
    }

    @include breakpoints.container(min, md, $offset: 1) {
      // your styles for containers larger than md
    }
  }
  ```

  ::

  ::step
  #headline
  Usage in code

  #default
  To access the breakpoints programmatically (e.g. inside a [ResizeObserver](https://developer.mozilla.org/en-US/docs/Web/API/ResizeObserver)), use the `ONYX_BREAKPOINTS` object:

  ```ts [Using breakpoints programmatically]
  import { ONYX_BREAKPOINTS } from "sit-onyx";

  console.log(`Width for sm breakpoint is: ${ONYX_BREAKPOINTS.sm}px`);
  ```

  ::

</steps>
